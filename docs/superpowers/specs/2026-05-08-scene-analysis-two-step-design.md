# SnapExtract 场景分析页两步重设计

日期：2026-05-08  
范围：`snapextract_v3.html` 的四个场景分析页  
状态：已确认设计，待实现

## 1. 目标

把当前场景分析页统一改成两步链路：

`Step 1 补充信息` → `Step 2 结果展示`

本次调整解决三个问题：

1. 分析页当前信息结构过散，评委第一眼看不清主链路。
2. 原文预览、右侧视图、结果弹窗并存，主结果不够直接。
3. 四个场景交互不统一，讲解成本高。

## 2. 已确认约束

1. 四个场景全部统一改，不只改论文。
2. 分析页不再显示原文。
3. 分析页拆成两步，不保留现有多层结果结构。
4. 第二步结果默认直接铺开，不使用结果弹窗。
5. 第一步补充信息完成后，直接自动进入第二步，不再要求手动点“开始分析”。
6. 页面保持单屏表达，主工作区尽量铺满，不走长滚动叙事。

## 3. 总体结构

### 3.1 页面骨架

四个场景统一采用同一个 `scene shell`：

- 顶部：沿用首页同款全局 `stepper`
- `stepper` 状态：直接高亮 `03 场景分析`
- 主体：单一分析容器，占满 `stepper` 下方的全部可用高度
- 场景内部状态：
  - `scene_step = supplement`
  - `scene_step = result`

不新增路由，不拆新页面，仍沿用：

- `#/scene/resume`
- `#/scene/contract`
- `#/scene/statement`
- `#/scene/paper`

### 3.2 状态切换

进入场景后默认先落在 `Step 1 补充信息`。

当用户完成当前场景所需补充动作后：

- 简历 / 论文：输入完成并确认后，直接进入结果态
- 合同 / 对账单：补充文件上传完成后，直接进入结果态

切换发生在当前页面内部，只做场景状态切换，不发生跳路由。

## 4. Step 1 设计

### 4.1 角色

`Step 1` 只承担一个职责：补足当前场景分析需要的上下文。

它不是结果页，也不是说明页，因此不放：

- 原文预览
- demo 说明文案
- 长段解释
- 额外视图列表

### 4.2 布局

`Step 1` 使用整页双区布局：

- 左侧：主补充区
- 右侧：主文件摘要区

其中：

- 左侧承载输入框或补充文件槽位
- 右侧只展示最小必要信息，例如主文件名、识别到的文档类型、当前场景名称

右侧只作为辅助，不承载正式分析结果。

### 4.3 场景映射

#### 简历

输入字段：

- 岗位名称 / 职级
- 核心要求
- JD 文本或职责摘要

#### 合同

补充内容：

- 对方合同 / 对比合同文件

#### 对账单

补充内容：

- 银行对账单文件

#### 论文

输入字段：

- 论文标题（可选）
- 研究方向
- 关注问题 / 使用目的

## 5. Step 2 设计

### 5.1 角色

`Step 2` 直接展示场景分析结果。

用户进入后，不再需要二次点开卡片，也不再通过弹窗查看核心结果。

### 5.2 布局

`Step 2` 使用 `2 x 2` 大结果网格：

- 四个主模块平铺满主区
- 页面本身不做长滚动
- 若单卡内容过长，只允许卡片内部滚动

每个结果模块都是完整信息卡，不再采用：

- 结果入口卡
- 点击查看结果
- 结果详情弹窗

### 5.3 视觉优先级

统一规则：

- 左上第一块始终是最重要的主结论或主信息
- 其余三块承接判断、建议、补充重点
- 内容默认展开，避免评委错过关键信息

## 6. 四个场景的结果模块

### 6.1 简历

四块结果：

1. 候选人信息
2. 匹配判断
3. 推进建议
4. 面试问题与待确认项

### 6.2 合同

四块结果：

1. 关键信息
2. 风险判断
3. 签署建议
4. 待谈判事项

### 6.3 对账单

四块结果：

1. 识别字段
2. 核对判断
3. 差异摘要
4. 验算指标

### 6.4 论文

四块结果：

1. 研究问题
2. 方法路径
3. 核心结果
4. 结论与可借鉴点

## 7. 现有模块调整

### 7.1 从主路径移除

以下内容不再出现在分析页主结构中：

- 原文 preview
- 左侧 preview rail
- 右侧 `view list`
- `paper-detail-modal`
- `scenario-detail-modal`
- 结果入口小卡片
- 旧 footer stepper / 旧 flow 承载
- baseline 在分析页主区的存在感

### 7.2 保留但降级

以下能力可以保留，但不能再主导布局：

- 论文公式翻转卡
- 论文增强能力
- 追问能力

这些能力只允许作为结果模块内的次级内容或次级入口出现，不能再占据独立主区域。

## 8. 交互规则

### 8.1 Step 1

- 简历 / 论文：用户完成输入后直接进入分析结果
- 合同 / 对账单：补充文件接入后直接进入分析结果
- 不保留单独的大型“开始解析”“重新解析”主按钮

### 8.2 Step 2

- 默认完整展示四块结果
- 不弹结果 modal
- 不需要“点击查看详情”
- 卡片内容必须直接可读

## 9. 实现建议

建议围绕现有 `renderScene` 做一次收口，不另起新路由：

1. 引入统一状态 `scene_step: 'supplement' | 'result'`
2. `renderResultMain` 改为根据 `scene_step` 输出两种主界面
3. `renderPreview`、`renderViewList`、`renderLayer3` 在分析页主路径中降级或清空
4. 把四个场景现有分散的结果 detail 数据，直接映射到四块主结果卡
5. 取消结果弹窗绑定逻辑，保留必要数据生成函数

## 10. 验收标准

### 10.1 通用

1. 四个场景都遵循同一套两步结构。
2. 分析页不显示原文。
3. 第二步结果默认直接铺开。
4. 不出现结果弹窗主路径。
5. 页面主工作区保持单屏表达。

### 10.2 场景

#### 简历 / 论文

- 用户填写补充字段后，自动进入结果页

#### 合同 / 对账单

- 用户补传文件后，自动进入结果页

#### 全场景

- 结果页固定为四块主结果
- 四块结果都无需再次点击才能看到主体信息

## 11. 风险与注意点

1. 现在场景逻辑里存在较多旧结构耦合，例如 preview、view list、detail modal、layer3，需要一起收口。
2. 若只隐藏旧模块但不移除绑定，容易留下空白区域或无效事件。
3. 第二步结果全部铺开后，文案长度必须重新压缩，否则会撑坏单屏布局。
4. 论文场景现有增强模块较多，必须明确它们退居次级，否则会重新把页面拉回复杂结构。

## 12. 本次不包含

本次设计不包含：

- 首页解析对照页改版
- 上传逻辑改版
- 新增后端能力
- 新增结果导出链路
- 新增场景外功能
